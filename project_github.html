Automating Deployments with GitHub CI/CD

Part 5 of the AWS Cloud Resume Challenge

January 28, 2026

Now that my website was live, secured with HTTPS, and delivered globally through CloudFront, the next step was improving the deployment process. Up to this point, every time I made a small change—whether updating my resume text or adjusting a line of CSS—I had to manually upload files to my S3 bucket through the AWS Console. While this works for small experiments, it quickly becomes inefficient and error-prone.

To solve this, I implemented a CI/CD pipeline using GitHub Actions. Now, whenever I push changes to my repository, GitHub automatically deploys the updated website to AWS. This approach reflects how modern development teams manage infrastructure and applications: through automation and version control rather than manual updates.

Step 1: Creating the GitHub Repository

The first step was moving my website into version control. I created a repository on GitHub and pushed my website files, including:

  • HTML
  • CSS
  • JavaScript
  • Images and other static assets
This repository now acts as the single source of truth for the project. Instead of keeping multiple versions of files locally, every change is tracked through Git commits. Version control also allows me to easily roll back changes, track the history of updates, and collaborate more effectively if the project grows in the future.

Step 2: Granting GitHub Permission to Deploy

For GitHub to deploy updates to AWS, it needs permission to interact with AWS services. To accomplish this, I created a dedicated IAM user in AWS with programmatic access. I then attached a policy allowing the minimal permissions required for deployment. The workflow requires access to:

  • s3:PutObject
  • s3:DeleteObject
  • cloudfront:CreateInvalidation
These permissions allow GitHub to upload updated files to the S3 bucket and invalidate the CloudFront cache when changes occur. After creating the user, I stored the credentials securely inside my GitHub repository using GitHub Secrets. The secrets I added were:
  • AWS_ACCESS_KEY_ID
  • AWS_SECRET_ACCESS_KEY
These secrets allow GitHub Actions to authenticate with AWS without exposing credentials in the repository code.

Security Note

Using access keys works well for small projects, but the modern best practice is to use OpenID Connect (OIDC) instead. OIDC allows GitHub to assume an AWS role and receive temporary credentials without storing long-lived access keys. While access keys were sufficient for this stage of the project, OIDC is the recommended approach for production environments.

Step 3: Creating the Deployment Workflow

GitHub Actions workflows are defined using YAML configuration files. Inside my repository, I created the following folder structure: .github/workflows/. Inside that folder, I added a file called: deploy.yml. This file defines the steps GitHub should run whenever new code is pushed to the repository.

The workflow is triggered automatically when code is pushed to the main branch. The pipeline performs the following steps.

  1. Checkout the Repository Code
    GitHub pulls the latest version of the repository so the workflow can access the website files.
  2. Configure AWS Credentials
    The workflow authenticates with AWS using the credentials stored in GitHub Secrets.
  3. Deploy the Website Files to S3
    The deployment uses the AWS CLI command: 
    aws s3 sync . s3://jonathanlayman.com --delete
    This command uploads the contents of the repository to the S3 bucket. The --delete flag ensures the bucket stays synchronized with the repository by removing files from S3 that no longer exist in the source code.
  4. Invalidate the CloudFront Cache
    Because CloudFront caches website files, the pipeline also runs a cache invalidation command: 
    aws cloudfront create-invalidation --distribution-id YOUR_DISTRIBUTION_ID --paths "/*"
    Without this step, visitors might continue seeing older cached versions of the site.

Step 4: Testing the Pipeline

After committing the workflow file, the CI/CD pipeline was automatically activated. To test the deployment process, I made a small change to my index.html file and pushed the update to GitHub. Inside the Actions tab of the repository, I could see the workflow executing in real time.

Once the pipeline completed successfully, a green checkmark confirmed that the deployment had finished. Within seconds of refreshing jonathanlayman.com, the changes were live on the website. This confirmed that the pipeline was successfully updating the S3 bucket and invalidating the CloudFront cache automatically.

Tips and Lessons Learned

Use Branch Protection

To reduce the risk of pushing broken code, I enabled branch protection rules on the main branch. This helps enforce good development practices as the project grows.

Cache Invalidation Matters

CloudFront aggressively caches content to improve performance. Without running a cache invalidation during deployments, visitors may continue seeing outdated files. Including this step in the CI/CD pipeline ensures users always receive the latest version of the site.

The Result

At this stage of the project, the website deployment process is now fully automated. Whenever I push changes to GitHub:
  • GitHub Actions runs the deployment workflow
  • The updated files are synchronized to the S3 bucket
  • CloudFront invalidates its cache
  • The updated site becomes available globally within seconds
This automation eliminates manual uploads and makes future updates significantly faster and more reliable.
