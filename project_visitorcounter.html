Building a Serverless Visitor Counter with Lambda and DynamoDB

Part 7 of the AWS Cloud Resume Challenge.

February 23, 2026
Frontend Website Code: https://github.com/jlayman09/Layman_CloudResumeWebsite
Backend Serverless Code (Lambda + DynamoDB): https://github.com/jlayman09/cloud-resume-backend

With my frontend fully secured and my cost monitoring in place, it was time to add the first dynamic feature to the website: a visitor counter.
This is one of the core components of the AWS Cloud Resume Challenge because it demonstrates how data moves through a modern cloud architecture. A request starts in the user's browser, calls an API endpoint, triggers backend logic, updates a database, and returns a result back to the frontend.

To build this feature, I implemented a serverless architecture using:
  • Amazon DynamoDB to store the visitor count
  • AWS Lambda to process the request and update the database
This approach eliminates the need to manage servers while still allowing the application to scale automatically.

Step 1: Setting Up the Database (DynamoDB)

The first step was creating a place to store the visitor count. For this, I used Amazon DynamoDB, AWS's fully managed NoSQL database. DynamoDB is designed for high availability and low latency, making it ideal for lightweight application data like a simple visitor counter.

I created a table called: cloud-resume-stats
The table was configured with the following key:
  • Partition Key: id (String)
This key uniquely identifies each item in the table. To initialize the counter, I manually created the first item with the following attributes:
  • id: visitors
  • count: 0
This provides the starting value that the Lambda function will increment each time the API is called.

Step 2: Writing the Backend Logic (AWS Lambda)

Next, I created an AWS Lambda function to handle the visitor counting logic. Lambda functions allow you to run code in response to events without provisioning or managing servers. The function executes only when it is triggered, which makes it cost-effective for small workloads like this.

I wrote the function in Python using the boto3 library, which is the AWS SDK for Python. The Lambda function performs three main tasks:
  1. Connect to the DynamoDB service
  2. Increment the visitor count stored in the table
  3. Return the updated value to the caller
The most important part of the function is the atomic update operation. Instead of reading the value and writing it back manually, the function uses DynamoDB's update_item operation with an update expression. This allows the database to handle the increment operation safely.

Why this matters: If multiple visitors load the website at the same time, DynamoDB ensures the counter increments correctly without race conditions or lost updates.

Step 3: Securing the Connection with IAM

By default, a Lambda function cannot access other AWS services. Permissions must be explicitly granted through IAM roles. Each Lambda function runs with an execution role, which defines what services the function is allowed to interact with.

To follow the Principle of Least Privilege, I created a custom policy that allows only the minimum required permissions. The policy grants the following actions:
  • dynamodb:GetItem
  • dynamodb:UpdateItem
These permissions are scoped specifically to the ARN of the cloud-resume-stats DynamoDB table. After attaching the policy to the Lambda execution role, I tested the function using the Lambda test feature to confirm it could successfully update the database.

Step 4: Creating an Endpoint for the Frontend

In order for the website to trigger the Lambda function, it needs an HTTP endpoint that the browser can call. For this stage of the project, I enabled a Lambda Function URL, which provides a simple HTTPS endpoint for invoking the function.

This allowed the frontend JavaScript to send requests directly to the Lambda function. Because the website is hosted on a different domain than the Lambda endpoint, I also needed to configure CORS (Cross-Origin Resource Sharing). CORS allows browsers to safely make requests between different domains.

To secure the endpoint, I configured the following settings:
  • Allowed Method: GET
  • Allowed Origin: https://jonathanlayman.com
By restricting the allowed origin to my domain, I ensured that only my website can call the Lambda function.

Step 5: Integrating the Counter with the Frontend

With the backend in place, the final step was connecting it to the website. I added a small JavaScript script to the frontend that runs when the page loads. The script performs three actions:

  1. Sends a request to the Lambda Function URL
  2. Waits for the JSON response containing the updated visitor count
  3. Updates the visitor counter displayed on the page
When a user visits the site:
  • The browser sends a request to the Lambda endpoint
  • Lambda increments the value stored in DynamoDB
  • The updated visitor count is returned to the browser
  • The page displays the updated number
This completes the full data flow from frontend to backend and back again.

Lessons Learned

This part of the project highlighted an important reality of backend development in the cloud: writing the code is often the easiest part. A significant portion of the work involves configuring permissions, security controls, and service integrations correctly.

Ensuring the IAM role permissions were tightly scoped and configuring CORS properly were essential steps to make the system both functional and secure.

The Result

At this point, the website now includes a fully functional serverless visitor counter. The architecture looks like this:

User Browser → Lambda Function URL → AWS Lambda → DynamoDB Table → Updated visitor count returned to the browser

Every time someone loads the page, the counter increments automatically.

What's Next?

Up to this point, most of the infrastructure for this project was created manually through the AWS console. This approach is commonly referred to as ClickOps. While this works for learning and experimentation, production environments rely on Infrastructure as Code (IaC) to ensure that infrastructure is reproducible, version-controlled, and easy to manage.

In the final stage of this project, I will begin converting my AWS infrastructure into code using Terraform, one of the most widely used Infrastructure as Code tools in the industry. This will allow me to manage the entire environment—S3, CloudFront, Lambda, and DynamoDB—through configuration files instead of manual setup.
