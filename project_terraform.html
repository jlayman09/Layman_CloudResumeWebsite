Rebuilding the Infrastructure with Terraform

Part 8 of the AWS Cloud Resume Challenge.

March 7, 2026

After completing the frontend and backend portions of the project using the AWS console, I reached an important realization: while manual configuration works for learning, it is not how infrastructure is managed in professional environments.

Modern cloud infrastructure is built using Infrastructure as Code (IaC). Instead of clicking through the AWS console, engineers define infrastructure in code so it can be version controlled, replicated, and deployed automatically.

To bring this project closer to real-world DevOps practices—and to prepare for the Terraform Associate certification—I migrated my existing AWS resources into HashiCorp Terraform and began managing the entire environment as code.

This process involved two major phases:
  • Importing my existing AWS resources into Terraform
  • Using Terraform to build new infrastructure for my contact form email service

Prerequisites

Before starting the migration, I prepared my development environment. Tools used:
  • Visual Studio Code
  • Terraform CLI
  • AWS CLI

The AWS CLI was configured with credentials from an IAM user that has permission to manage the resources used in the project. This setup allows Terraform to authenticate with AWS and manage infrastructure programmatically.

Phase 1 - Importing Existing Infrastructure

My website was already live and functioning. Instead of deleting those resources and rebuilding them, I used Terraform Import. Terraform Import allows existing AWS resources to be brought under Terraform management without recreating them.

Step 1.1 - Initializing the Terraform Project

I created a new project directory and added a file called main.tf. Inside this file, I defined the AWS provider so Terraform knows which platform it should interact with.

Example configuration: 
terraform {
 required_providers {
   aws = {
     source  = "hashicorp/aws"
     version = "~> 5.0"
   }
 }
}

provider "aws" {
 region = "us-east-1"
}
This configuration tells Terraform to use the AWS provider and operate within the us-east-1 region, where several of my resources were created.

Step 1.2 - Creating Resource Skeletons

Next, I created placeholder resource blocks for the infrastructure that already existed. These blocks initially contained very little configuration because Terraform would later import the actual resource settings.

Example resource definitions:

resource "aws_s3_bucket" "website_bucket" {
}

resource "aws_dynamodb_table" "stats_table" {
}

These skeleton resources act as placeholders so Terraform can map them to the real AWS resources.

Step 1.3 - Importing the AWS Resources

With the resource blocks defined, I used the Terraform CLI to import the existing infrastructure. Each resource was imported using its unique identifier.

Example commands: 
terraform import aws_s3_bucket.website_bucket jonathanlayman.com
						terraform import aws_dynamodb_table.stats_table cloud-resume-stats

These commands connect the Terraform configuration to the existing AWS resources.

Step 1.4 - Reconciling Terraform State

After importing the resources, Terraform still did not fully understand their configuration. To fix this, I ran terraform plan repeatedly. Terraform compares the configuration files to the real infrastructure and displays any differences.

My job was to update the .tf files until Terraform reported: "No changes. Your infrastructure matches the configuration." This step ensured Terraform's configuration accurately reflected the real AWS environment.

Phase 2 - Building New Infrastructure with Terraform

Once the existing infrastructure was under Terraform management, I began creating new resources entirely through code. The next feature I wanted to build was a serverless email service for the contact form on my website. This service allows visitors to submit a message directly through the site.

Step 2.1 - Configuring Amazon SES

To send emails from the contact form, I used Amazon Simple Email Service (SES). Terraform was used to create the SES domain identity for my domain.

Example resource:

resource "aws_ses_domain_identity" "website_email" {
 domain = "jonathanlayman.com"
}

This allows AWS to verify that my domain is authorized to send email through SES.

Step 2.2 - Creating IAM Permissions

The Lambda function responsible for sending email needs permission to use SES. Using Terraform, I created an IAM role and policy that allows the Lambda function to call the SES send email API. Following security best practices, the policy grants only the specific permissions required for the function.

Step 2.3 - Deploying the Email Lambda Function

Next, I deployed the Lambda function responsible for processing messages submitted through the website contact form. Terraform packages the Lambda code and uploads it automatically. The function receives the form submission, processes the request, and sends an email using Amazon SES.

Step 2.4 - Creating the API Endpoint

For the contact form to communicate with the Lambda function, the website needs a public endpoint. I created this using API Gateway, which acts as the entry point for HTTP requests. Using Terraform, I configured an API Gateway endpoint that accepts POST requests and forwards them to the Lambda function. This architecture allows the contact form to securely trigger the backend email service.

Terraform Best Practices Learned

Working through this process reinforced several important Terraform practices.
  • Remote State: Terraform state files contain sensitive information about infrastructure. I configured a remote backend using an S3 bucket to store the state file securely.
  • Git Ignore Rules: To prevent sensitive information from being exposed, I added the following to my .gitignore file:
    • terraform.tfstate
    • terraform.tfvars
  • Outputs: Terraform output blocks allow useful information—such as API endpoints or CloudFront URLs—to be displayed automatically after deployment. These outputs help streamline testing and integration.

Final Architecture

At the end of this process, my project infrastructure is now fully managed using Terraform. The architecture includes:
  • S3 - Static website hosting
  • CloudFront - Global CDN and HTTPS
  • Route 53 - DNS management
  • Lambda - Serverless compute
  • DynamoDB - Visitor counter database
  • API Gateway - Backend API endpoints
  • SES - Email delivery service
All of these resources can now be deployed or recreated using Terraform configuration files.

Conclusion

Learning Terraform fundamentally changed how I think about infrastructure. Instead of manually configuring services through the AWS console, the entire environment is now described in code. This makes the infrastructure easier to manage, easier to reproduce, and far more scalable.
If this environment ever needed to be rebuilt, the entire architecture could be redeployed in minutes using Terraform.
Completing this final stage marks the end of my AWS Cloud Resume Challenge. What started as a simple static website evolved into a fully automated, serverless cloud architecture managed entirely through Infrastructure as Code.
